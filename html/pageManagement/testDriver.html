Attributes - Parameters - Tokens - Examples
Test Driver
com.perfectoMobile.page.keyWord.step.AbstractKeywordStep
The Test Driver is a project that ties all libraries together for a powerful test execution environment. The project allows for execution of test using pure XML without any java editor or IDE. When executing the Test Driver a property file is passed in that allows for full cofnigruation of the test run.
Driver Configuration
driver.frameworkType:Specifies which driver specification to use. Valid values are
  • XML: A provider based off of XML
driver.configName:Specifies the name of the file used for the provider XML
driver.propertyAdapters:Specifies a comma separated list of fully qualified class names implementing the PropertyAdapter interface. Property Adapters allow for customer set of properties to be configurable via the test driver configuration file
driver.personas:A comma separated list of persona names. If supplied this will enabled the Wind Tunnel subsystem and execute the test cases against each named persona
driver.enableCaching:Enables the client side smart caching sub-system. in application with may static check this can help to increase performance by caching page objects during successive calls
driver.deviceInterrupts:A set of interrupts that are randomly executed during a test. That valid values are
  • CALL: Initiates a device phone call being received during a test
  • SMS: Simulates a text being received during a test case
driver.validateConfiguration:Validates all configuration and reports which test and devices it would run against, but does not execute any tests
Cloud Configuration
Clouds are collections of devices or browsers used for testing
cloudRegistry.provider:Specifies which cloud configuration provider to use. Valid values are
  • XML: A provider based off of XML
  • CSV: A provider based off of a comma separated list of values
  • EXCEL: A provider using Excel
cloudRegistry.fileName:Specifies the name of the file used for the providers XML, EXCEL and CSV
cloudRegistry.tabName:Specifies the name of the tab used for the EXCEL provider
cloudRegistry.cloudUnderTest:The name of the cloud being used for this test
Application Configuration
Applications are items being tested
applicationRegistry.provider:Specifies which application configuration provider to use. Valid values are
  • XML: A provider based off of XML
  • CSV: A provider based off of a comma separated list of values
  • EXCEL: A provider using Excel
applicationRegistry.fileName:Specifies the name of the file used for the providers XML, EXCEL and CSV
applicationRegistry.tabName:Specifies the name of the tab used for the EXCEL provider
applicationRegistry.applicationUnderTest:The name of the cloud being used for this test
Device Configuration
Devices are pulled from the specified cloud and used for testing
deviceManagement.provider:Specifies which device configuration provider to use. Valid values are
  • RESERVED: This will use the specified cloud and search for any devices that are reserved to the user specified in your cloud configuration
  • AVAILABLE: This will use the specified cloud and search for any devices currently available
  • NAMED: Allows you to select specific named devices
  • XML: A provider based off of a devices defined via XML
  • CSV: A provider based off of a comma separated list of values
  • EXCEL: A provider using Excel
deviceManagement.driverType:Specifies the underlying driver type to be used for this test. Value values are:
  • PERFECTO: This will use the Perfecto Mobile Driver
  • APPIUM: Uses the Appium drivers (IOS/ANDROID) for testing Native application on Mobile devices
  • WEB: Use the RemoteWebDriver for test web based application on Mobile Devices and Desktop Browsers
deviceManagement.fileName:Specifies the name of the file used for the providers XML, EXCEL and CSV
deviceManagement.tabName:Specifies the name of the tab used for the EXCEL provider
deviceManagement.deviceList:A comma separated list of device identifies used to define mobile device for use in the NAMED provider

pageManagement.siteName:Specifies the name of the site under test. When using the EXCEL provider this name is also corresponds to the tab name contains the page element data for this site
Page Data
Page data is used to provide data driven testing

  • XML: A provider based off of XML
  • EXCEL: A provider using Excel
pageManagement.pageData.fileName:Specifies the name of the file used for the providers XML and CSV
pageManagement.pageData.tabNames:Specifies a comma separated list of tab names defining each of the record types for the EXCEL provider. Each tab represents a single named record type.
Content Management
Content Management is used to provide varying content for application testing to support resource file and multi lingual applications
pageManagement.content.provider:Specifies which content management configuration provider to use. Valid values are
  • XML: A provider based off of XML
  • EXCEL: A provider using Excel
pageManagement.content.fileName:Specifies the name of the file used for the providers XML and CSV
pageManagement.content.tabName:Specifies the tab name containing the page content for the EXCEL provider.
pageManagement.content.keyColumn:A zero based index of the column containing the lookup key in the EXCEL provider.
pageManagement.content.lookupColumns:A comma separated list of zero based indexes of all lookup columns in the EXCEL provider.
Artifacts and Reporting
Artifacts are details from a test run that are saved for later review
artifactProducer.parentFolder:Specifies the parent folder in which all test artifacts are stored. This will will contain a folder for each test run using the run time. Contained will be a folder for ever test case run and then a folder for each device that it was run on
artifactProducer.storeImages:One working with client side imaging this flag specifies if the images are stored locally. This is used for the WCAG report
artifactProducer.imageLocation:Specifies which folder to store images in of the storeImages flag is set to true
artifactProducer.logLevel:Specifies the detail that is written to the Console log when using the CONSOLE_LOG artifact. Validvalues are
  • FATAL: Severe errors that cause premature termination. Expect these to be immediately visible on a status console
  • ERROR: Other runtime errors or unexpected conditions. Expect these to be immediately visible on a status console
  • WARN: Use of deprecated APIs, poor use of API, 'almost' errors, other runtime situations that are undesirable or unexpected, but not necessarily "wrong". Expect these to be immediately visible on a status console
  • INFO: Interesting runtime events (startup/shutdown). Expect these to be immediately visible on a console, so be conservative and keep to a minimum
  • DEBUG: detailed information on the flow through the system
  • TRACE: More detailed information on the flow through the system
artifactProducer.automated:A list or artifacts that are automatically created at the appropriate times. These will be stored in the specified out folder for test artifacts. Valid artifact types are:
  • EXECUTION_REPORT_PDF: Gets a copy of the Perfecto Execution report in PDF format
  • EXECUTION_REPORT_HTML: Gets a copy of the Perfecto Execution report in HTML format
  • EXECUTION_REPORT_CSV: Gets a copy of the Perfecto Execution report in CSV format
  • EXECUTION_REPROT_XML: Gets a copy of the Perfecto Execution report in XML format
  • FAILURE_SOURCE: Extracts the state of the device and application. That states is extracted and saved as failureDom.xml to allow for quick debugging of object identification issues
  • WCAG_REPORT: If WCAG operations were used (Branding, contrast, etc) then this will contain the extracted images and their operational statistics.
  • DEVICE_LOG: Extracts the previous 1000 lines of the device's log file
  • CONSOLE_LOG: Writes the console output for the specified test on the specified device. Only information pertaining to that test/device is stored here
  • EXECUTION_RECORD_HTML: Contains a clean execution report of the test steps that were executed. In addition, a master index file will be created showing the status of all tests executed in this run
  • EXECUTION_RECORD_CSV: Contains an CSV based reports of all test steps that were executed
  • EXECUTION_TIMING: Contains a small timing report of elements marked as timed
