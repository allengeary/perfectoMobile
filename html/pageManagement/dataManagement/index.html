Data Driven Testing
com.perfectoMobile.page.data.PageDataManager
Data-driven testing (DDT) is a term used in the testing of computer software to describe testing done using a table of conditions directly as test inputs and verifiable outputs as well as the process where test environment settings and control are not hard-coded. Data-driven testing is the creation of test scripts to run together with their related data sets in a framework. The framework provides re-usable test logic to reduce maintenance and improve test coverage. Input and result data values can be stored in one or more central data sources. The data comprises variables used for both input values and output verification values. The DDT framework performs "play back" of harvested data producing a powerful automated regression testing tool. While a test may use multiple tables from the data source, only one data source may be used
Usage
Once configured page data can be accessed via the PageDataManager singleton class via one of the following methods
getRecord( recordType ):Retrieves a single record from the named record type. The records are stored in a FIFO (first in first out) queue and each used record as added back to the end of the queue for reuse. If the recordType had the locked flag set then this record will not be available again until putPageData is called with this pageData object. The PageData object that was returned will have access to all of the named fields
getRecords( recordType ):Retrieves all records from the named record type. All records will be available and the locking logic is ignored
XML Configuration
When defining page data using XML the data designer will create record type elements. A record type element is the same concept as a database table in that it contains a set of records.
name:The name attribute represents the data source name as it will be referenced within the test cases
lockrecords:Lock records is a flag (true/false) indicating if a record can be used in parallel. When true a record will be locked by an individual iteration of a test case and released for reuse upon completion otherwise the records can be reused as requested by the test case
The individual contained record elements make up the test data. Record items may contains any number of named attributes comprising the data in addition to the elements specified below. The same named attributes/elements must exist throughout all record in the same record type and they will be available for use in the test case. If you want to create a tiered data structures where page data can contain other page data structures then the pipe character will be used. When surrounding a value with the pipe character it indicates that this value is actually a reference to another table. So |authData| indicates that this field is a reference to the authData record type and will contain all of the rows. You can filter the data as well by providing a set of criteria, each contained within a set of square brackets as needed. |authData:[type='user'][age='10']| indicates that this references the authData record type however only the fields that contains the field value of user for type and 10 for age will be selected
name:The name attribute represents the field name as it will be referenced within the test cases
active:Active is a flag (true/false) indicating if this record is active. Setting a record's active flag to false tells the subsystem to ignore it during a run
